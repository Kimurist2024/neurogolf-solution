# Proposal 003: タスク別ゴルフ工場(Lane T 本格稼働)

- 提案者: Fable(メインセッション)
- 実装者: Codex(ハーネス)+ LLM ゴルフワーカー(Opus/Codex 混成)
- 前提: 公式ベスト 6349.47。公開カーネルからの収穫は尽きた(merge 採用ゼロを確認済み)
- 根拠: `docs/research/discussion-methods-recent.md` / `cost-gap-analysis.md` / `dtype-feasibility.md`

## 1. 戦略(リサーチ結論の適用)

直近 3 週間の Discussion 分析で確定した現メタ:
- トップ(7400〜7700)は **NN ではなくゼロパラメータの純関数型テンソルプログラム**をタスクごとに手書き/LLM 合成している
- 証言: タスク別 LLM ゴルフループで **エージェント 1 本 8 時間 +10〜15 点**、ChatGPT Plus のみで 7600 到達例
- 我々のギャップは全域に分布(中央値コスト 15,230 → 目標数百)。**1 タスクずつ潰すしかない**

## 2. アーキテクチャ

```text
[キュー] cost 降順の 400 タスク (docs/research/cost-gap-analysis.json)
   ↓
[ブリーフ生成] タスクごとに golf ブリーフ (グリッド可視化 + 現ネット解剖 + コスト目標)
   ↓
[ゴルフワーカー] 1 タスク = 1 LLM エージェント
   numpy で変換を再実装 → onnx.helper で最小グラフ化 → ハーネスで検証 → 反復 (失敗制約を蓄積)
   ↓
[ゲート] gold 全例一致 + マージン(生出力に 0〜0.25 なし) + 現ベストより厳密に安い
   ↓
[マージ] artifacts/handcrafted/ → merge_external.py --source で安全統合 → 改善ごとに提出
```

## 3. Codex への実装依頼(ハーネス 3 点)

### 3.1 `scripts/golf/brief.py --task N`
`docs/golf/briefs/taskNNN.md` を生成:
- train/test/arc-gen(先頭 3 件)のグリッドをテキスト描画(0-9 の数字グリッド、入出力並記)+ サイズ情報
- 現ベストネットの解剖: cost / memory / params / ノード数 / op ヒストグラム
- 目標コスト(900 と 314)と達成時スコア
- arc-gen 残り件数と検証コマンド

### 3.2 `scripts/golf/try_candidate.py --task N --onnx PATH`
候補 ONNX を一発判定:
- 公式バリデータ相当(ファイルサイズ / 禁止 op / 静的形状 / opset domain)
- gold 全例一致(train+test+arc-gen)、失敗時は**最初の不一致例の期待/実際グリッドをテキスト表示**(ワーカーの修正材料)
- マージンチェック(生出力の (0, 0.25) 帯)
- cost 算出と現ベスト比較
- 合格なら `artifacts/handcrafted/taskNNN.onnx` へ昇格(既存 handcrafted より安い場合のみ上書き)

### 3.3 `scripts/golf/status.py`
キュー進捗一覧: タスクごとの incumbent cost / handcrafted cost / 期待ゲイン残

実装規約: 既存 `scripts/lib/scoring.py` を再利用(スコアリング再実装禁止)。ファイル 800 行以内。型注釈。

## 4. ワーカー運用(Fable がオーケストレーション)

- パイロット: 優先キュー上位 5 タスク(255, 101, 133, 158, 096)に各 1 エージェント
- ワーカーへの指示テンプレ: ブリーフ + 「numpy 参照実装 → 等価な最小 ONNX」の手順 + try_candidate の使い方 + 失敗ログの蓄積
- パイロットで歩留まり(成功率・平均ゲイン・トークン消費)を計測してからフリート規模を決定
- 提出は従来どおり: マージ → ローカル期待スコア改善 → 即提出

## 5. 期待値

- 優先 50 タスクを cost 900 まで: **+235**
- 全 310 タスク(cost>2000)を cost 2000 まで: **+761**
- パイロット 5 タスクだけでも cost 900 到達なら **+13.5**

## 6. リスクと対策

- ローカル合格 ≠ グレーダー合格(プラットフォーム差)→ マージンゲート + 必要なら単一タスク zip での実測(1 日 100 提出枠を活用)
- arc-gen 過適合 → 全例検証は必須、ホールドアウトはワーカーに任せない(ハーネスが強制)
- ワーカーの幻覚 ONNX → try_candidate が唯一の真実。合格ファイル以外は採用経路なし

## 7. Amendment 1(2026-06-12、パイロット事故 + ユーザー並行実験の知見統合)

### 7.1 例フィット禁止(task096 事故、-15.4 点で実証)
公開例にフィットした決定木ネットは**ローカル全合格でも非公開ベンチで 0 点**。ワーカー規程:
- **ジェネレータ仕様(inputs/arc-gen-repo/tasks/)の変換を実装する。例データへのフィッティングは禁止**
- 最終報告で「仕様由来 / 例フィット由来」を自己申告。例フィット疑いは単一タスク zip で実測してからバンドル

### 7.2 一般化チェックリスト(ユーザー並行実験の知見)
採用してよい構成要素: connected components / BFS(有界展開)/ bounding box / 対称性 / 線分検出・補完 / 色ヒストグラム / オブジェクト関係 / copy・translate・extend / マスク変換。
禁止: 特定座標への固定出力、特定色への依存(タスク仕様が色を固定する場合を除く)、例の個数・配置の暗記、入力サイズ固定、オブジェクト数固定、一部の例だけ通る分岐。

### 7.3 グレーダー処理エラーの前歴タスク
task054 / task158 は別系統の提出で「Error processing onnx networks」を起こした前歴あり(ユーザー報告)。
このタスクの差し替えは慎重に: 採用前に公式バリデータ相当チェックを厳格適用し、不安があれば過去にグレーダーを通った版を優先する。

### 7.4 量子化の扱い
完全一致が要求されるため、出力が 1 セルでも変わる量子化は不採用。dtype 縮小は「閾値後マスク一致 + マージン」を通る場合のみ。

### 7.5 タスク単位ベスト選抜の原則(確立済み)
zip 単位ではなく**タスク単位**で全候補源(過去 zip・並行実験の成果物・handcrafted)から最良を選ぶ。
マージの現職コストは run レポートではなく**実ファイルの再スコア**を使う(stale 上書き事故防止)。

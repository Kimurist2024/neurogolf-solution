# Agent ループ 単一タスク改良 — 基本ルール(標準テンプレート)

> 【ユーザー厳命 2026-06-29】これは agent ループで **1 タスクを改良するときの基本ルール**。
> このセッション以降、単一タスク改良を回すときは必ず本ルールに従う。
> 関連: [BANNED_STRUCTURES.md](BANNED_STRUCTURES.md) / [ONNX_GOLF_PLAYBOOK.md](ONNX_GOLF_PLAYBOOK.md) / [ERROR_PATTERNS.md](ERROR_PATTERNS.md)

実行時の差し込みパラメータ(プレースホルダ):

- **現在の全体スコア**: `(今の最新スコア)`
- **対象タスク**: `task(任意のタスク)` ← 1 ループ = 1 タスクに集中

---

## 1. 目的(最優先順に)

1. **対象タスクのスコアを最低 +2 改善する**。可能なら **cost 5 以下** を目標。
2. **必ず「現状より改善した ONNX」を出力する**。理想 cost に届かなくても、現状より良い候補が得られたら必ず ONNX として保存する。**改善 ONNX を出さずに終了しない**。
3. 既存 ONNX の軽い最適化で改善が難しい / 途中で行き詰まったら、**必ず 1 から再構築**する(後述 §5)。

> score(task) = max(1, 25 - ln(cost))。cost を e 倍(約 2.718x)削ると +1 点。+2 を狙うなら **約 7.4x の cost 削減**が目安。小手先の数%削減で止めない。

---

## 2. 採用しないもの(検証時間の上限)

- **モデルが重すぎて検証できない状態は採用不可**。ローカル検証・公式デコード・fresh 検証を含め、**実用上 60 秒以内**に実行できる構成にする。
- cost が低くても、**検証時間が極端に長い ONNX は採用しない**(採用ゲート §6 の検証は **30 秒以内**に完了すること)。
- 指数的に遅くなる構造・過剰な Einsum・巨大な全探索・不要なブロードキャスト・全グリッドへの重い `Where`/`Expand`/`Scatter`/`Conv`・大量定数による力技は避ける。低 cost でも測定が長引くなら採用せず、**ROI / 座標ベース / ルールベースの直接生成**へ切り替える。

---

## 3. 使ってはいけない構造([BANNED_STRUCTURES.md](BANNED_STRUCTURES.md) に従う)

- **公式バリデータが必ず弾く(絶対禁止)**: `Loop` `Scan` `NonZero` `Unique` `Script` `Function`
- **経験則で追加禁止**: `Compress`、すべての `*Sequence*` 系、ネストグラフ(サブグラフ)、動的形状、`sparse_initializer`
- 全テンソルは**静的形状**。opset ドメインは **`''`** および **`ai.onnx`** のみ。`Equal` は **opset≥11**。
- ONNX ファイルは **1.44MB 以下**。
- **TopK は一律禁止ではない**。ただし **ローカル ORT 1.24 の `ORT_DISABLE_ALL` で採点が最後まで完走する構成のみ**採用。op 名だけで判断せず、**必ず実際のローカル grader の結果**を基準にする(opset18 TopK は通る、opset11 TopK は `NOT_IMPLEMENTED` の実証あり)。

---

## 4. ONNX 入出力契約

- 入力は 1 つ、名前 `input`、dtype/形状 **float32 [1,10,30,30]**(色 0..9 の one-hot、境界外は zero-hot)。
- 出力は 1 つ、名前 `output`、**完全静的形状**。
- スコアラは raw 出力を `raw > 0` で閾値化。閾値後マスクが期待 one-hot と**完全一致**(真の出力グリッド外のゼロも含む)。
- on セルは明確な正値、off セルは正確/非正値。**raw 値を (0, 0.25) に置かない**(環境差で閾値反転するため)。

---

## 5. 改良の手段(問わない / 行き詰まったら必ず再構築)

既存 ONNX の最適化、1 からの再構築、画像化した正解例からの規則推論、Python solver による検証、ルールベース生成、遺伝的アルゴリズム、いずれでも可。
ただし **小手先で改善が難しい / 途中で詰まったら、必ず 1 から再構築**する:

1. `inputs/arc-gen-repo/tasks/task_<hash>.py`(+ `common.py`)= 正典 generator 仕様を読む。可視例は**デバッグ用フィクスチャ**に過ぎない。
2. 必要に応じて train・test・arc-gen を**画像化**し、変換規則を考察する。
3. 規則の numpy solver を書き、train・test・arc-gen 完全一致 + fresh インスタンスで検証する(**例フィットではなく仕様をコンパイル**。過去に public-fit ネットが約 -15 点の私的事故を起こした)。
4. **最小構造の静的形状 ONNX** へ翻訳(最小 dtype、per-channel マスク、早期クロップ、Conv-bias 折り込み、output = Equal / paint-onto-input)。

---

## 6. 採用ゲート(以下を **すべて満たす場合のみ**採用)

1. `scripts/lib/scoring` を **`require_correct=True`** で実行し、**train・test・arc-gen が全て完全一致**し、**cost が算出できる**こと。
2. **公式 `neurogolf_utils` デコーダ**でも train・test・arc-gen の **wrong=0** を確認できること。
3. 生の float 出力に **(0, 0.25) の境界セルが存在しない**こと(環境差の閾値反転が起きない)。
4. generator が生成する **fresh データ**について **k>0** かつ **失敗 0 件**であること。**fresh で 1 件でも失敗した ONNX は採用しない**。
5. **検証全体が 30 秒以内**に完了すること。

### 採用前の ONNX 構造チェック(追加で必須)

- `onnx.checker` および **shape inference** を通過すること。
- initializer・attribute・tensor shape の整合性を確認すること。
- **Conv を使う場合は weight・bias・出力チャネル数が完全に一致**していること(bias 長 < out_ch のフリップ事故を防ぐ)。
- **ZIP の並び順だけで採点結果が変化する候補は採用しない**。構造上の問題がないことを確認する。

> 上記ゲートの主要部分は `scripts/verify_fix.py --task N --onnx PATH --k 30` が機械的に実施する(fresh k=30 / lib gold / official gold / margin)。これに通った上で、構造チェックと 30s/60s 時間制約を別途確認する。

---

## 7. 最終成果物(必ず出力)

- **改善済み ONNX**
- **改善内容の要約**
- **上記 §6 の 5 項目の検証結果**
- **ONNX 構造チェック結果**
- 改善済み ONNX を既存提出物へ差し替えた**提出用フル ZIP**(ファイル名は厳密に `submission.zip`)

改善できなかった場合は**その旨を明確に記載**する。ただし可能な限り 1 から再構築し、現状より改善した ONNX を出力することを最優先とする。

## 8. 報告の誠実性(厳守)

- **未実施の検証を実施済みと記載しない**。**推測を事実として報告しない**。
- 実際に確認した内容と未確認の内容を**明確に区別**して報告する。
- provenance を宣言する(spec 由来か否か)。spec 由来でないなら**汎化を主張しない**。
